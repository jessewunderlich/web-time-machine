## The 80/20 filter

Before listing ideas, here's the lens I'm applying:

- **What moves someone from "cute, closed tab" to "I sent this to three friends"?** → content depth + one unforgettable moment per era.
- **What keeps the site relevant in 18 months when 2021 stops being "recent"?** → Era 2026+ (agentic web), plus a mechanism for timeline to auto-extend.
- **What unlocks SEO / discovery that per-era OG cards hint at but don't yet earn?** → per-era landing pages that are actually worth landing on (not just bouncing).
- **What would a historian vs. a designer vs. a developer each want to cite?** → citations, sources, deep-dive content, embeddable widgets.

Anything that doesn't serve one of those four goals is skipped.

## Tier A — Ship next (highest leverage, ≤1 day each)

### A.1 Era 8: 2026+ (The Agentic Era) — the single highest-leverage item

**Why first:** The site currently ends in 2021 with "AI chatbots exist." That was news in 2022. In 2026 the frontier is agents-in-tool-use (Claude Code, Cursor agents, Devin, ChatGPT Agent Mode) and that's genuinely a different design surface than chat — which is the whole point of the site. Ending on 2021 makes the site feel **stuck in 2023**.

**Scope (deliberately minimal):**
- New `Era2026.tsx` after `Era2021`.
- Browser chrome: generic future browser (Dia? Comet? pick one real or make a composite).
- Content: one interactive element — a fake "agent running a task" that shows a terminal-style command log ("reading file…", "calling tool…", "writing 3 files…") with realistic latency. No real LLM — just a scripted sequence on scroll.
- Include real-2026 facts: MCP, Claude Code, Cursor, Devin, Operator/ChatGPT Agent Mode.
- Subtitle: "When the browser stopped being the destination."
- Add matching `/share/2026` OG card.

**Why cheap:** Reuses every existing primitive (BrowserChrome, HistoricalSites, EraTransition, SoundEngine stub, OG card). All the hard infra is there.

**Effort:** 4–6 hours including content writing + one OG card + one new era CSS module.

**Risk:** Content-honesty risk. 2026 isn't over yet — we need to describe what's happening, not predict. Frame as "mid-2026 snapshot" not "the future."

### A.2 Era 0: 1989–1990 (Pre-web) as a cold open

**Why:** Right now the site opens on a terminal with the ASCII art. Strong, but it opens in medias res — the web already exists. Adding a pre-web screen gives a **narrative arc**: before → birth → evolution.

**Scope:**
- New `Era1989.tsx` before `Era1991`. Pure CRT-green text on black (Xerox Alto / DEC VT100 aesthetic).
- Content: Tim Berners-Lee's March 1989 "Information Management: A Proposal" memo excerpts (public domain per CERN).
- No HistoricalSites (there are none — the web didn't exist).
- Fake CRT scanlines + phosphor glow via CSS.
- Transition into 1991: the CRT "boots" from prehistoric to Lynx.

**Why cheap:** The CRT aesthetic is one CSS module (`filter: contrast + green tint + scanline pseudo-element`). Content is public-domain. No historical sites to research.

**Effort:** 3–4 hours.

**Risk:** Gets the site to 9 eras. Bento grids, number-key nav (1–9), and URL anchors all scale without friction.

### A.3 Analytics — install Vercel Web Analytics + Speed Insights

**Why:** Right now we're shipping blind. We know Lighthouse is 97; we don't know whether anyone reaches era 2021, whether the soundtrack button is ever toggled, or whether mobile visitors bounce at era 2000's Flash popup. **These answers would reshape Tier B priorities.**

**Scope:**
- `npm i @vercel/analytics @vercel/speed-insights`
- Two one-liners in `layout.tsx`: `<Analytics />` and `<SpeedInsights />`.
- Add one custom event: `track('era_entered', { era })` on IntersectionObserver enter.
- Free tier covers what we need; no cookie banner required (Vercel Analytics is cookieless and GDPR-compliant).

**Why cheap:** 20 minutes from start to deployed.

**Effort:** 30 minutes.

**Payoff:** Unblocks data-driven decisions on everything in Tier B. Single best ROI item on this list.

### A.4 A "?" keyboard shortcut help modal

**Why:** KeyboardNav hint flashes for 3 seconds then hides, `aria-hidden="true"`. Users who miss it have zero way to discover shortcuts. Press-`?`-for-help is a web convention (GitHub, Vercel dashboard, Linear, every modern SaaS).

**Scope:**
- On `?` keypress anywhere (outside inputs), open a centered modal.
- Modal lists: arrows (adjacent era), 1–9 (jump to era), Home/End (jump to first/last), Esc (close any open overlay).
- Proper focus trap, focus-visible on close button, Esc dismisses, ARIA `role="dialog"` with `aria-modal="true"`.
- Show a tiny persistent hint at corner: `press ? for shortcuts` (instead of the current auto-hiding banner).

**Effort:** 1.5 hours.

**Payoff:** Discoverability + a11y both improve. Mobile users get nothing out of this (no keyboard) which is fine — desktop power-users are the audience for keyboard nav.

### A.5 Real historical screenshots (via Wayback Machine, on-demand, not iframed)

**Why:** `HistoricalSites` currently uses CSS-art placeholders. Charming, but Jesse's fact-check pass proved the educational value depends on **real** artifacts. A CSS-art "Yahoo 1997" is a cartoon. A real Yahoo 1997 snapshot is a citation.

**Scope (refined since prior roadmap):**
- **NOT** iframes (Wayback Machine is being increasingly rate-limited, blocked, and content-restricted — Guardian, NYT, others blocking scrapers since May 2025). Iframes would also tank our LCP.
- **YES** static screenshots stored as WebP/AVIF in `/public/historical/era-YYYY/site-slug.webp`. Generated once via a script that hits `https://web.archive.org/web/TIMESTAMP/URL` and screenshots with Playwright at 1200×800.
- Each card shows: screenshot + attribution line (`via archive.org · captured YYYY-MM-DD`).
- Hover/focus on the screenshot: badge appears "View on Wayback Machine ↗" linking to the canonical Wayback URL.
- `next/image` with `placeholder="blur"` for LCP.

**Why on-demand not iframe:**
- Iframes of Wayback are slow (cold-start 3–8s), load 3–5MB each, can be blocked by origin site robots.txt, and Wayback reserves the right to disable embeddings at any time.
- Static screenshots are 50–80KB WebP each, cacheable forever, immune to Wayback policy changes.

**Effort:** ~6 hours. 1 hour tooling (script), 4 hours running + curating captures for ~20 sites, 1 hour wiring into HistoricalSites.

**Licensing:** Wayback Machine captures are generally fair-use for educational/archival contexts with attribution. Not legal advice — if Jesse wants to be maximally safe, attribute every capture ("Captured by Internet Archive, YYYY-MM-DD, original URL") and honor takedown requests.

**Payoff:** Makes the site genuinely citable in academic / journalism contexts, which is the SEO + authority multiplier.

## Tier B — Meaningful depth (1–2 day each)

### B.1 Per-era "Learn More" drawer

**Why:** Current site is vibes-first. Users who want depth have nowhere to go. A side drawer is the low-scope way to give them a path without cluttering the main scroll.

**Scope:**
- `[+] Learn More` pill at the top-right of each era.
- Opens a right-side drawer (or full-screen overlay on mobile).
- Content per era: 4–5 paragraph long-form history + timeline of key events + 3–5 external links (Wikipedia, webdesignmuseum.org, Wayback Machine, contemporary blog posts from archive.org).
- Uses `<details>`/`<summary>` semantics so it works without JS and is SR-friendly.
- URL-bindable: `/#era-2000/details` opens drawer on load.

**Effort:** 1.5 days. The long pole is writing the content — 7–9 eras × 1500 words each, sourced + citeable.

**Payoff:** The scroll-through user gets vibes. The depth-seeker gets an essay. Each era becomes a landing page worth linking to.

**Decision gate:** Only worth building if analytics (A.3) confirm people scroll to the later eras. If 80% bounce by Era 2000, the drawer is wasted work.

---

### B.2 Embeddable era iframes (growth mechanic)

**Why:** Right now the site is a destination. Embeddable widgets turn it into **infrastructure** — every web-history blog post becomes a distribution channel.

**Scope:**
- New route: `/embed/era/[era]` — strips nav/progress bar, renders a single era in a 600×900 responsive frame.
- `<iframe src="https://web-time-machine-coral.vercel.app/embed/era/1996" width="600" height="900" style="border:0" title="1996: GeoCities era" allow="clipboard-write" sandbox="allow-scripts allow-same-origin"></iframe>`
- Add CSP `frame-ancestors *` (or explicit allowlist) so third-party sites can embed.
- Add a tiny "powered by Web Time Machine →" link in the corner (growth loop).
- On the main site, add an "Embed this era" button that reveals the copy-paste snippet (modal-style).

**Effort:** 6–8 hours. 2h for the embed route, 2h for the copy-snippet UI, 2h for testing in real contexts (Medium, dev.to, Notion, Substack), 2h for CSP + docs.

**Payoff:** Authority backlinks. If 20 dev bloggers embed this in "how I got into web dev" posts, that's 20 domain-authority-high backlinks pointing to our pages. This is how Embed.ly and CodePen became defaults.

---

### B.3 Shareable bookmark-slash-progress state

**Why:** Deep-link to era is solved. But **deep-link to a specific moment in an era** (e.g., "the GeoCities under-construction gif is what got me into web design") is not.

**Scope:**
- Add ScrollTrigger-aware deep-link support for sub-sections within an era. URL format: `/#era-1996/webring` or `/#era-1996/counter`.
- Each era exports a `sections: { [key]: string }` map of anchor points → scroll position fractions.
- Native `scrollIntoView` to that nested anchor on load (waits for hydration).

**Effort:** 6 hours.

**Payoff:** Fine-grained shareability. Enables things like "check out this hit counter: `web-time-machine.vercel.app/#era-1996/counter`" in a Slack channel. Low-cost flex.

---

### B.4 Content expansion: 2 real-artifacts per era replacing synthetic ones

**Why:** Currently most era bodies are Jesse's prose describing the era. Prose is infinite; artifacts are scarce. Artifacts are memorable.

**Scope per era:**
- 1991: embed an excerpt of the actual `info.cern.ch/hypertext/WWW/TheProject.html` HTML source (not a dramatization).
- 1996: embed a real archived GeoCities homepage (Wayback screenshot, not simulated).
- 2000: embed a real surviving Flash game (play via Ruffle emulator — `ruffle-selfhosted` is MIT-licensed, works in-browser, ~300KB WASM).
- 2005: embed a real archived MySpace profile (Wayback screenshot).
- 2010: embed the actual iOS 7 app store CSS recreated exactly (not a flat-gradient approximation).
- 2015: embed a real Material Design spec page excerpt from Google's archive.
- 2021: embed an actual chatgpt.com screenshot from Dec 2022 (pre-redesign).

**Effort:** Variable per era — 30min to 4h each. Total ~1 day.

**Payoff:** Shifts the site from "someone's creative vision of web history" to "a curated archive with author voice." These are different products. The latter is more linkable + citable.

**Risk:** Ruffle (Flash emulator) is ~300KB WASM — load lazily, gate behind a "click to play" button, only in Era2000.

## Tier C — Growth / distribution (days-weeks, high variance)

### C.1 Product Hunt launch

**Why:** PH launches still drive 3k–20k visitors in 24 hours for well-positioned projects. Web-design history is a broadly-interesting-niche topic.

**Pre-reqs:** A.3 (analytics to measure the spike), A.1 + A.2 (broader era coverage makes it feel complete), and a 60–90 second screen recording demo.

**Tactics:**
- Launch on a Tuesday at 12:01am PT (PH ranking starts fresh).
- Title: `Web Time Machine — scroll through 35+ years of web design, styled in each era` (the "styled in each era" is the hook).
- Hunt with someone who has 500+ followers if possible, but ship anyway if not.
- First comment from maker: 2 paragraphs on **why** you built it. Jesse's angle: "learning web design meant learning its history; there was no resource that showed me how eras *felt*, not just what they looked like."
- Pin 3 comments: feature request, factoid, technical choice explanation. Respond to every single comment in first 4 hours.

**Effort:** 4 hours prep + 12 hours day-of engagement.

**Payoff:** Variable. Median outcome ~3k visits + 40–100 upvotes. Best case: a Vercel / GitHub / dev.to blog post gets triggered by the PH ranking → long tail.

---

### C.2 Dev.to / Hashnode long-form post — "The architecture behind Web Time Machine"

**Why:** Dev.to remains the best platform for dev-interest long-form. Posts on "how I built X" with real code samples still rank top 3 for the right keywords.

**Scope:** 2000–3000 word post. Sections:
1. Why I wanted the site to *feel* different per era (not just look).
2. GSAP ScrollTrigger + pinned spacers (why Lenis didn't work).
3. The contrast problem during era transitions (luminance averaging bug → panel solution).
4. Satori OG image generation at build time.
5. View Transitions API for era jumps.
6. Lessons about authenticity vs. modernity (the tension that defines this project).

**Effort:** 1 day writing + 2 hours code excerpts + publishing.

**Payoff:** Evergreen backlinks. Each post drives 200–800 views, most of which click through to the live site. More importantly: **signals Jesse as a named author** in the "interactive storytelling" niche.

### C.3 HN "Show HN" post

**Why:** Show HN can drive 10k–200k visitors in a good day. Success rate is low (≤20% front page) but the distribution is right-tail: when it hits, it hits hard.

**Pre-reqs:** All Tier A done, strong site title meta, mobile perfect, load time <2s global.

**Tactics:**
- Post on a Wednesday 10am ET.
- Title: `Show HN: Web Time Machine – scroll through 35 years of web design, styled in each era`
- First comment: short, technical, and honest about trade-offs. HN values candor over marketing.
- Respond to the top 3 comments within 15 minutes.

**Effort:** 2 hours prep (post draft, checking title claims, making sure the server can handle bursts), 6+ hours engagement window.

**Vercel autoscales, but the landing page must load <1.2s P90 or HN bounces hard. Speed Insights data (A.3) would tell us if we're at risk.**

### C.4 Academic citation push

**Why:** "Web design history" is taught in HCI courses, MFA programs, digital humanities. If one syllabus cites this site, it becomes durable linkbait.

**Tactics:**
- Identify 20 professors teaching web history (via syllabi search on OpenSyllabus).
- Send a personal email: "I built this, thought you might find it useful as a supplement. Totally free, no ads."
- No pitch — just share. The honest ones will link if they like it; the useful ones will give feedback.
- Work offers: "I'll add any era-specific deep links you want for your curriculum — just tell me what topics."

**Effort:** 2–3 hours outreach + variable follow-up.

**Payoff:** If 2 of 20 cite it, that's 2 `.edu` backlinks — which do disproportionate work for authority.

### D.2 Auto-regenerate OG cards when content changes

**Current behavior:** OG cards are SSG'd at build time. If Jesse fixes a fact in `ERA_CARDS.tagline`, the OG image updates on next deploy. Good.

**Opportunity:** Stable preview URLs for PRs (Vercel does this automatically but only for the HTML page; the OG image URL includes a content hash so social caches bust correctly). This already works — just documenting the behavior so it's not accidentally broken by a well-meaning refactor.

**Effort:** 0 hours. Document only.

### D.3 RSS feed for timeline updates

**Why:** The site will grow — Era 2026+, corrections, new historical screenshots. An RSS feed of "what's new on Web Time Machine" is free subscriber-retention, and `atom.xml` feeds remain a stealth growth channel in dev circles.

**Scope:** `/feed.atom` route that reads `CHANGELOG.md` and outputs Atom XML.

**Effort:** 3 hours including a CHANGELOG discipline.

**Payoff:** Low traffic, but the users who subscribe via RSS are the highest-intent users (devs, historians).

## Tier E — Deliberately skipped (recorded here so they don't reappear)

These came up in prior roadmaps or research; flagging as **not worth building right now** with reasoning.

### ✗ Dark mode toggle
**Why skip:** The site is 7 eras, each with intentional palette. A sitewide dark/light override would destroy the point. User's OS preference can already affect the splash screen + nav chrome — that's enough.

### ✗ Multi-language
**Why skip (for now):** Non-English web history is genuinely a richer story (Japanese 2000s keitai web, Chinese Web 2.0, Latin American dev scene). But proper i18n is a 2-week project. Jesse's stated constraint is "ship things". This waits until the English version is proven worth translating.

### ✗ Backend / contribution form
**Why skip:** Fully static is a feature. Moving to a backend adds moderation, spam, abuse, cost, and maintenance. If contribution is wanted, a GitHub Issues link ("suggest a correction") solves it for free.

### ✗ "Build your own era" sandbox
**Why skip (for now):** Scope risk is enormous. The current site is finite + curated; a builder turns it into infinite + user-generated, which is a different product. Worth considering in 6 months once the core site is proven.

### ✗ Podcast / narrated tour
**Why skip (for now):** Recording 7 × 2-minute episodes is a day. Editing is another day. Distribution (podcast feeds, embedding) is another day. Low ROI until audience size justifies the cost of maintenance.

### ✗ Bundle-size micro-optimizations (ROADMAP §4.1 "polyfill stripping")
**Why skip:** Current mobile perf is 97. Shaving 14KB wins 0.02 in Lighthouse, does nothing for real-user metrics (we're network-bound, not parse-bound on modern devices). Not worth the browserslist churn risk.

## Decision framework going forward

Any future "should we build X?" question should answer these four:

1. **Does it improve memorability?** (people send it to friends)
2. **Does it improve relevance?** (the site doesn't feel stuck in a past year)
3. **Does it improve discoverability?** (people outside our network find it)
4. **Does it improve depth?** (returning visitors find new things)

If an idea doesn't move at least one of these four needles meaningfully, it's polish — which is fine, but not urgent. This site has already gotten more polish rounds than most shipped sites ever see.
